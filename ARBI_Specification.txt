1. Introduction 
1.1. Purpose 

The Acquired Resource Base Indictor (ARBI) format is a binary serialization specification designed for representing semantic knowledge graphs 
extracted from web resources. It optimizes for 
a. storage efficiency (70-85% smaller than JSON)
b. Parsing speed (10-50x faster than JSON)
c. Memory locality (cache-friendly layout)
d. Streaming Support (process without full loading)

1.2 Scope 

This document specifies 
 a. Binary Layout and Daa Structures 
 b. Encoding and decoding algorithms 
 c. Data type definitions 
 d. Schema constraints 
 e. Implementation requirements 


1.3 Terminology 

Term   Definition 
Node    A knowledge entity (document, concept, relationship)
ARBI Score  Composite metric indicating resource acquisiton value
Dictionary  Interned string table for compression 
Streaming   Processing data in chunks without full memory load 
Zero-copy   Direct mmap access without deserialization 

2. Design Goals & Requirements 
2.1 Primary Goals 

Goal      Target      Measurement 
Size Efficiency    <=30% of JSON      Bytes per knowledge node 
Parse Speed      >=10x JSON      Nodes per second 
Memory Efficiency     <= 40% of JSON      RAM per node 
Streaming support      fixed overhead      Chunk independence 

2.2 Non-Goals 
- Human readability (debugging tools provided separately)
- self-describing schemas (schema registry assumed)
- cross-platform floating point consistency (IEEE 754)

2.3 Constraints 
This section talks about constraints 
- Maximum node size : 16 MB (enables 32-bit offsets)
- Maximum dictionary entries : 4294967295( 2^32-1)
- Maximum relationships per node: 65535
- Maximum tags per node:255
- Maximum attribute value size: 2^32-1 bytes 

3. Format Overview 

3.1 File Structure 

The ARBI format comprises of 4 key sections 
a. Heaber with 32 fixed bytes 
b. Dictionary which comprises of string interning table of variable size 
c. Node section which is of variable length and is sequential in nature 
d. index section which provides random access lookup table of variable size 


---------------------------------------
|          HEADER                      |
|       (32 bytes fixed)
---------------------------------------
---------------------------------------
|          DICTIONARY                      |
|       (String interning table)
|        variable size
---------------------------------------

---------------------------------------
|          NODE Section                      |
|       (variable-length nodes)
|        sequential
---------------------------------------

---------------------------------------
|          INDEX SECTION                     |
|       (Random access lookup table)
|        variable size
---------------------------------------

3,2 Data Organization Principles 
1. locality: related data stored contiguously 
2. Alignment: 1-byte packaing (no padding except where noted)
3. Endianness: little-endian (LE) througout 
4. Variable Encoding: LEB128 for integers < 2^64


3.3 Byte Order 
All multi-byte integers are using little-endian byte order (ARM, x86_64, RISC-V). Applcations on big-endian systems (SPARC) must perform byte swapping during decoding phase.

4.1 Header Structure 

Size: 32 bytes fixed 

Struct arbi_header_t{
  uint8_t magic[4];          //0x41 0x52 0x42 0x49 ("ARBI")
  uint16_t version;          //0x0001 for v1.0
  uint8_t flags;             // Bitflags (Table1)
  uint8_compression;         // Compression codec (Table2)
  uint32_t dict_offset;      // Bye offset to dictionary section 
  uint32_t nodes_offset;     //Byte offset to node section 
  uint32_t index_offset;     //Byte offeset to index section

4.3 Node Section

Each node is variable-length and may be preceded by alignment padding only in streaming mode.
c

struct arbi_node_header_t {
    uint32_t id;               // Local node ID (0 to total_nodes-1)
    uint8_t flags;             // Node flags (Table 3)
    uint16_t name_dict_id;     // Dictionary ID for entity name (16-bit)
    uint8_t num_tags;          // Number of tags (0-255)
    // Followed by num_tags dictionary IDs (variable-length encoded)
    // Followed by arbi_score (float32)
    // Followed by relationship section (arbi_relationship_section_t)
    // Followed by attribute section (arbi_attribute_section_t)
    // Followed by optional embedding (arbi_embedding_t)
};

struct arbi_relationship_section_t {
    uint16_t num_relationships;    // Number of relationships (0-65535)
    uint32_t relationships_offset; // Offset to relationship array
    // Relationships data at offset (variable length)
};

struct arbi_attribute_section_t {
    uint16_t num_attributes;       // Number of attributes (0-65535)
    uint32_t attributes_offset;    // Offset to attribute array
    // Attributes data at offset (variable length)
};

4.6 Embedding Structure
c

struct arbi_embedding_t {
    uint16_t dimensions;           // Number of dimensions (1-65535)
    uint8_t quantization_bits;     // 4 or 8 bits per dimension
    uint8_t scale_factor;          // Scaling factor (stored as uint8)
    // Followed by compressed embedding data:
    // if quantization_bits == 8: dimensions * int8_t
    // if quantization_bits == 4: (dimensions + 1) / 2 * uint8_t (packed)
    // if quantization_bits == 0: dimensions * float32 (uncompressed)
};

5. Data Types & Encoding
5.1 Variable-Length Integer (LEB128)

ARBI uses LEB128 (Little Endian Base 128) for unsigned integers.

Encoding Algorithm:
python

def encode_uleb128(value: int) -> bytes:
    if value < 0:
        raise ValueError("ULEB128 requires unsigned integer")
    
    result = []
    while True:
        byte = value & 0x7F
        value >>= 7
        if value == 0:
            result.append(byte)
            break
        result.append(byte | 0x80)
    return bytes(result)

Decoding Algorithm:
python

def decode_uleb128(data: bytes, offset: int) -> tuple[int, int]:
    value = 0
    shift = 0
    pos = offset
    
    while True:
        if pos >= len(data):
            raise ValueError("Incomplete ULEB128")
        
        byte = data[pos]
        pos += 1
        value |= (byte & 0x7F) << shift
        
        if (byte & 0x80) == 0:
            break
        
        shift += 7
        if shift > 63:
            raise ValueError("ULEB128 exceeds 64 bits")
    
    return value, pos - offset

5.4 String Encoding

    All strings: UTF-8 encoded (no BOM)

    No null terminators (length explicitly stored)

    Dictionary IDs use variable-length encoding

5.5 Floating Point

    IEEE 754 binary32 for float32

    IEEE 754 binary64 for float64

    No special handling for NaN/Inf (allowed but discouraged)

6.2 ARBI Score Calculation

The ARBI score quantifies the acquisition value of a resource.
python

def calculate_arbi_score(resource: dict) -> float:
    """
    Calculate ARBI score from resource indicators.
    
    Formula: 
        ARBI = 0.4*A + 0.3*F + 0.2*C + 0.1*S
    
    Where:
        A = Authority (domain rank)
        F = Freshness (content update frequency)
        C = Cross-reference (backlink quality)
        S = Structure (format richness)
    """
    
    # Authority: logarithmic domain rank
    domain_rank = resource.get('domain_rank', 10**6)
    A = 1 - (min(domain_rank, 10**6) / 10**6)
    
    # Freshness: sigmoid of days since update
    days_since_update = resource.get('days_since_update', 30)
    F = 1 / (1 + math.exp(5 * (days_since_update/30 - 0.5)))
    
    # Cross-reference: normalized backlinks
    backlinks = resource.get('backlink_count', 0)
    C = min(backlinks / 1000, 1.0)
    
    # Structure: format richness (0-1)
    has_schema = resource.get('has_schema_org', False)
    has_opengraph = resource.get('has_opengraph', False)
    has_jsonld = resource.get('has_json_ld', False)
    S = (has_schema + has_opengraph + has_jsonld) / 3
    
    return 0.4*A + 0.3*F + 0.2*C + 0.1*S

7.1 Conformance Requirements

MUST implement:

    Header parsing and validation

    Dictionary decoding

    Node iteration (sequential)

    LEB128 encoding/decoding

    CRC32 checksum verification

SHOULD implement:

    Index section for random access

    Compression support (ZSTD recommended)

    Streaming mode

    Attribute array/object nesting

MAY implement:

    Zero-copy mmap access

    SIMD acceleration for parsing

    Custom dictionary preloading

7.2 Error Handling

Implementations MUST handle:
Error	Detection	Recovery
Invalid magic bytes	Header validation	Reject file
Checksum mismatch	CRC32 comparison	Reject file
Incomplete LEB128	Bounds check	Error with position
Out-of-range dictionary ID	Validate < num_entries	Error, skip node
Offset overflow	Check file bounds	Error, stop parsing
3	GZIP	Gzip compression
4-255	RESERVED	Future codecs

8.3 Denial of Service Prevention

    Bomb resistance: Limit nesting depth to 64 for arrays/objects

    Allocation guard: Check total_size against available memory

    Timeouts: Implement read timeouts for streaming

8.4 Privacy Considerations

ARBI files may contain sensitive data. Implementations SHOULD:

    Provide encryption wrapper specification

    Support secure memory wiping

    Implement access controls via indices
